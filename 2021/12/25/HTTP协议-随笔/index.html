HTTP协议-随笔

透视 HTTP 协议

超链接文档系统三项关键技术：

  • URI: 即统一资源标识符，作为互联网上资源的唯一身份
  • HTML: 即超文本标记语言，描述超文本文档
  • HTTP：即超文本传输协议，用来传输超文本。

URI 和 URL的区别？

互联网初期
20世纪60年代，美国国防部高等研究生计划署(ARPA)建立了ARPA网，被认为是如今互联网的始祖

HTTP/0.9
HTTP/0.9 是于1991年提出的 –> 主要是用于学术交流 需求是非常简单的 以纯文本格式交流
90 年代初期：结构简单，页面内容只可读。所以只允许 GET 请求 并且在响应请求之后立即关闭连接

HTTP/1.0
1996 年正式发布
由于多媒体的发展(例如：图片格式、MP3音乐格式的出现)，开始准备往HTTP里添加各种特性

  1. 增加了 HEAD、POST 等新方法 —> 注意：POST 请求是在1.0版本才出现的…
  2. 增加了响应状态码，标记可能的错误原因
  3. 引入了协议版本号概念
  4. 引入了 HTTP Header（头部）的概念，让 HTTP 处理请求和响应更加灵活；
  5. 传输的数据不再仅限于文本。

HTTP/1.1
1999 年发布

  1. 增加了 PUT、DELETE 等新的方法；
  2. 增加了缓存管理和控制； (缓存管理是啥？ 控制是啥？)
  3. 明确了连接管理，允许持久连接； (持久连接在HTTP/1.1中是默认开启的，所以你不需要专门为了持久连接去HTTP请求头设置信息，如果你不想要采用持久连接，可以在HTTP请求头中加上 Connection:close 目前浏览器中对于同一个域名，默认允许同时建立6个TCP持久连接)
  4. 允许响应数据分块（chunked），利于传输大文件
  5. 强制要求 Host 头，让互联网主机托管成为可能(提供虚拟主机)(一台物理主机上绑定多个虚拟主机，每个虚拟主机都有自己的单独的域名，这些单独的域名都公用同一个IP地址，因此，HTTP/1.1的请求头中增加了Host字段，用来表示当前的域名地址，这样服务器就可以根据不同的Host值做不同的处理)。
  6. 引入了客户端 Cookie 机制

HTTP/1.1依旧存在的问题
队头阻塞问题依旧严重
TCP的慢启动 —> 不太懂…
同时开启了多条TCP连接，那么这些连接会竞争固定的带宽 —> 这是一个细节
安全性问题

HTTP/2

2015 年发布 –> 以 google SPDY 协议为基础，综合其它多方的意见推出HTTP/2

由于 HTTPS 已经在安全方面做的非常好了，所以 HTTP/2 的唯一目标就是改进性能

高度兼容HTTP/1.1

HTTP/2 对于 HTTP的兼容性非常强(降低了学习成本~) –>
HTTP/2 把 HTTP 分解成了 语义 和 语法 这两个部分，语义层不做改动，与HTTP/1完全一致(例如：请求方法、URI、状态码、头字段等都没有变化)
甚至 HTTP/2 没有在 URI 里引入新的协议名，仍然用 HTTP 表示明文协议,用 HTTPS 表示加密协议 –> 让用户也是平滑过渡

在语义保持稳定不变前提下，HTTP/2在语法层做了巨大的改变。完全更改了HTTP报文的传输格式(不再是纯文本，而是二进制协议)

HTTP/2的变化

头部压缩

利用专门研发的算法：HPACK 进行报文 Header压缩 –> 压缩率：50%~90%
因为在语义上 HTTP/2 和 HTTP/1是兼容的，所以报文还是有 Header + Body 构成的，但是在请求发送前，必须要用 HPACK 算法来压缩头部数据
HTTP/2 由于压缩头部，流量损耗会进一步降低

二进制格式

报文由纯文本格式 转变为 二进制格式：非常方便计算机的解析 精确度上升，速度上升
头部数据压缩之后，HTTP/2 就要把报文拆成二进制的帧准备发送
原来的 Header + Body 报文结构 –打散为–> 多个 「二进制帧(HEADERS帧存放头数据、DATA帧存放实体数据)」 “碎片”

虚拟的「流」

流是二进制帧的双向传输序列
以流(虚拟的)的方式传递 二进制帧
HTTP/2 可以在一个TCP连接上用 流 同时发送多个「碎片化」的消息 –> 多路复用：多个往返通信都复用一个连接来处理
(注意：虽然帧是乱序收发，但是同一个流里都有相同的流ID)
流中的帧不是无序的，有着严格的先后顺序

在流的层面上看，消息是一些有序的帧序列，而在连接的层面上看，消息却是乱序收发的帧。多个请求/响应之间没有了顺序关系，不需要排队等待，也就不会出现队头阻塞问题，降低了延迟，大幅度提高了连接的利用率

强化安全

互联网上通常所能见到的 HTTP/2 都是使用 HTTPS 协议名，跑在 TLS 1.2+上面, 禁用了很多不安全的密码套件
h2: 表示加密的HTTP/2
h2c: 表示明文的 HTTP/2， 多出的字母c的意思是clear text

细节问题

HTTP/2.0 的普及率不算太高， 很多网站是 HTTP/1.1

HTTP/2 把 头部压缩 作为性能改进的一个重点

HTTP/2 的消息不再是 Header+Body 的形式，而是分散为多个二进制帧(请求发送前会把Header + Body,进行头部压缩之后拆解为二进制的帧进行发送)

问题：HTTP/2 的大文件上传该怎么操作呢？

把简单的系统变复杂，要比把复杂的系统变简单容易得多

HTTP/3

2018 年发布 –> 基于 google 的 QUIC 协议
在HTTP/2的基础上实现了质的飞跃，真正完美地解决了队头阻塞问题
基于UDP

HTTP 是什么？

HTTP: 超文本传输协议 –> 拆成三个部分： 超文本、传输、协议
协议： 一种约束、规范，遵守协议则就要遵守这些约束和规范
传输： A 传给 B，B 传给 A。HTTP 是双向协议。 这里恰好可以用客户端、服务端的数据传送来说明
超文本：所谓超文本，字面意思就是超越文本的文本。它是文字、图片、视频、音乐等多媒体的混合体，并且它拥有超链接。能够从一个超文本跳跃到另一个超文本

URI 和 URL的区别？

URI 和 URL 都能够指定唯一的一个资源。但是URI的宗旨是 无论你用的是什么方式，只要你能唯一的确定一个资源，就都算URI的一种实现，而URL就是用定位的方式(…/…/…/)来定位具体的资源
URI = URL + URN

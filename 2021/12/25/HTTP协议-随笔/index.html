HTTP协议-随笔

透视 HTTP 协议

互联网初期
20世纪60年代，美国国防部高等研究生计划署(ARPA)建立了ARPA网，被认为是如今互联网的始祖
20世纪70年代，基于对ARPA网的实践和思考，研究人员发明出了著名的TCP/IP协议。

1989年，一个老头 伯纳斯·李 发表了一篇论文，这篇论文提出了在互联网上构建超链接文档系统的构想。
他在这篇论文中提到了三项关键技术 超链接文档系统三项关键技术：

  • URI: 即统一资源标识符，作为互联网上资源的唯一身份。 (注意：uri和url的区别)
  • HTML: 即超文本标记语言，描述超文本文档。
  • HTTP：即超文本传输协议，用来传输超文本。

HTTP/0.9
HTTP/0.9 是于1991年提出的 –> 主要是用于学术交流 需求是非常简单的 以纯文本格式交流
90 年代初期：结构简单，页面内容只可读。所以只允许 GET 请求(从服务器里获取HTML文档) 并且在响应请求之后立即关闭连接 (无法持久连接)
(当时只有一个请求行，没有请求头和请求体。 返回的文件内容是以ASCII字符流来传输的因为都是HTML格式的文件，所以用ASCII字节码传输比较合适)

HTTP/1.0 –> 具备多种类型资源的接受和方法
1996 年正式发布
由于多媒体的发展(例如：图片格式、MP3音乐格式的出现)，开始准备往HTTP里添加各种特性

  1. 增加了 HEAD、POST 等新方法 —> 注意：POST 请求是在1.0版本才出现的…
  2. 增加了响应状态码，标记可能的错误原因
  3. 引入了协议版本号概念
  4. 引入了 HTTP Header（头部–> 响应头和请求头）的概念，让 HTTP 处理请求和响应更加灵活；
  5. 传输的数据不再仅限于文本ASCII码格式(支持多媒体格式)。
  6. 为了缓解服务器的压力，在HTTP/1.0中提供了**Cache机制(Cache-control?)**，用来缓存已经下载过的数据

HTTP/1.0的方案是通过请求头和响应头来进行协商，来判断该文件将返回什么类型的文件、该采用什么形式的压缩方式，提供什么语言的文件，以及文件的具体编码方式：

问题：队头阻塞问题
HTTP 1.0 请求与请求之间是串行的，如果我发送了A请求，那么A请求的响应返回之前，你的B请求不管多么着急都出不去。如果A就是请求卡在那，B请求也就只能堵着不能动
HTTP 1.1 使用管线化来解决这个问题：允许多个HTTP请求批量地提交给服务器，不过尽管发送动作可以并行，但是服务器依然需要根据请求顺序来回复浏览器的请求。本质上还是存在队头阻塞问题
HTTP 2 利用多路复用，一次连接建立后，只要这个连接还在，那么客户端就可以在一个链接中批量发起多个请求，并且请求之间不堵塞，请求之间高度独立，实现真正并行请求

HTTP/1.1 –> 支持长连接、虚拟主机、引入cookie
1999 年发布，HTTP/1.1是对HTTP/1.0的小幅度修正，但有一个重要的区别是：它是一个正式的标准

  1. 增加了 PUT、DELETE 等新的方法；
  2. 增加了缓存管理和控制； (缓存管理是啥？ 控制是啥？) –> 这里是指的CDN吗？
  3. 明确了连接管理，允许持久连接； (持久连接在HTTP/1.1中是默认开启的，所以你不需要专门为了持久连接去HTTP请求头设置信息，如果你不想要采用持久连接，可以在HTTP请求头中加上 Connection:close 目前浏览器中对于同一个域名，默认允许同时建立6个TCP持久连接，这样减轻了服务器额外的负担)
  4. 允许响应数据分块（chunked），利于传输大文件
  5. 强制要求 Host 头，让互联网主机托管成为可能(提供虚拟主机)(一台物理主机上绑定多个虚拟主机，每个虚拟主机都有自己的单独的域名，这些单独的域名都公用同一个IP地址，因此，HTTP/1.1的请求头中增加了Host字段，用来表示当前的域名地址，这样服务器就可以根据不同的Host值做不同的处理)。
  6. 引入了客户端 Cookie

HTTP/1.1依旧存在的问题
队头阻塞问题依旧严重
TCP的慢启动 —> 不太懂…
同时开启了多条TCP连接，那么这些连接会竞争固定的带宽 —> 这是一个细节
安全性问题

HTTP/2 –> 性能优化 核心就是多路复用

2015 年发布 –> 以 google SPDY 协议为基础，综合其它多方的意见推出HTTP/2

影响HTTP/1.1效率的三个主要因素
HTTP/1.1对于带宽的利用率很低。比如100M带宽，实际的下载速度只能到达12M/s,加载速度也就2M/s左右。出现这些问题的原因如下：

  • TCP慢启动：刚开始建立好TCP连接后TCP会采用一个很慢的速度去发送数据，后面才慢慢加快，这个过程其实就是慢启动 (这其实也是为了减少网络拥塞的一种策略)
  • 同时开启多条TCP连接会竞争这固定的带宽： 并且有点TCP连接下载的是一些关键资源，例如html、css文件，而有些是图片、视频资源，但是多条TCP连接无法选择哪些关键资源优先下载
  • 队头堵塞问题

而多路复用能解决上述的三个问题：
细谈多路复用：
HTTP/2的思路是一个域名只使用一个TCP长连接来传输数据，这样整个页面资源的下载过程只需要一次慢启动，也避免了多个TCP连接竞争带宽所带来的的问题。并且HTTP/2实现了资源的并行请求，不需要一个资源请求完才能进行下一个资源的请求。可以把请求分成一帧一帧的数据去传输，这样带来了一个额外的好处，就是当收到一个优先级高的请求时，比如收到JS或者CSS关键资源的请求，服务器可以暂停之前的请求来优先处理关键资源的请求(利用的就是二进制分帧，并且这些分帧都带有ID编号，服务器接收到这些帧后，会将所有相同ID的帧合并为一条完整的请求信息。)

注意：对于开发者而言，其实是没有感知变化的。我们依然可以使用cookie来保持登录状态，使用cache来缓存本地文件，真正变化的是传输方式，而开发者其实是感受不到传输变化的

二进制分帧层还能够实现请求的优先级、服务器推送、头部压缩等特性。从而大大提升了文件传输效率

由于 HTTPS 已经在安全方面做的非常好了，所以 HTTP/2 的唯一目标就是改进性能

高度兼容HTTP/1.1

HTTP/2 对于 HTTP的兼容性非常强(降低了学习成本~) –>
HTTP/2 把 HTTP 分解成了 语义 和 语法 这两个部分，语义层不做改动，与HTTP/1完全一致(例如：请求方法、URI、状态码、头字段等都没有变化)
甚至 HTTP/2 没有在 URI 里引入新的协议名，仍然用 HTTP 表示明文协议,用 HTTPS 表示加密协议 –> 让用户也是平滑过渡

在语义保持稳定不变前提下，HTTP/2在语法层做了巨大的改变。完全更改了HTTP报文的传输格式(不再是纯文本，而是二进制协议)

HTTP/2的变化

头部压缩

利用专门研发的算法：HPACK 进行报文 Header压缩 –> 压缩率：50%~90%
因为在语义上 HTTP/2 和 HTTP/1是兼容的，所以报文还是有 Header + Body 构成的，但是在请求发送前，必须要用 HPACK 算法来压缩头部数据
HTTP/2 由于压缩头部，流量损耗会进一步降低

请求优先级

HTTP/2提供了请求优先级，可以在发送请求时，标上该请求的优先级，这样服务器接受到请求之后，会优先处理优先级高的请求

二进制格式，不再是纯文本

报文由原来的纯文本格式 转变为 二进制格式：非常方便计算机的解析 精确度上升，速度上升
头部数据压缩之后，HTTP/2 就要把报文拆成二进制的帧准备发送
原来的 Header + Body 报文结构 –打散为–> 多个 「二进制帧(HEADERS帧存放头数据、DATA帧存放实体数据)」 “碎片”

虚拟的「流」

流是二进制帧的双向传输序列
以流(虚拟的)的方式传递 二进制帧
HTTP/2 可以在一个TCP连接上用 流 同时发送多个「碎片化」的消息 –> 多路复用：多个往返通信都复用一个连接来处理
(注意：虽然帧是乱序收发，但是同一个流里都有相同的流ID)
流中的帧不是无序的，有着严格的先后顺序

在流的层面上看，消息是一些有序的帧序列，而在连接的层面上看，消息却是乱序收发的帧。多个请求/响应之间没有了顺序关系，不需要排队等待，也就不会出现队头阻塞问题，降低了延迟，大幅度提高了连接的利用率

服务端推送

在 HTTP1.x 中，如果用户请求了资源 A，结果发现自己如果要用资源 A，那么必须依赖资源 B，这时他不得不再消耗一个请求。
HTTP2.0 中，允许服务器主动向客户端 push 资源。也就是说当服务器发现客户端请求了资源 A，却忘了请求资源 A 依赖的资源 B 时，它可以主动将资源 B 顺手推送给客户端，没必要再消耗一个请求

强化安全

互联网上通常所能见到的 HTTP/2 都是使用 HTTPS 协议名，跑在 TLS 1.2+上面, 禁用了很多不安全的密码套件
h2: 表示加密的HTTP/2
h2c: 表示明文的 HTTP/2， 多出的字母c的意思是clear text

细节问题

HTTP/2.0 的普及率不算太高， 很多网站是 HTTP/1.1

HTTP/2 把 头部压缩 作为性能改进的一个重点

HTTP/2 的消息不再是 Header+Body 的形式，而是分散为多个二进制帧(请求发送前会把Header + Body,进行头部压缩之后拆解为二进制的帧进行发送即先压缩再拆解)

问题：HTTP/2 的大文件上传该怎么操作呢？

把简单的系统变复杂，要比把复杂的系统变简单容易得多

HTTP/3

2018 年发布 –> 基于 google 的 QUIC 协议
在HTTP/2的基础上实现了质的飞跃，真正完美地解决了队头阻塞问题
基于UDP来实现的

对于TCP连接，可以看成是一个按照顺序传输数据的管道，管道中的任意一个数据丢失了，那之后的数据都需要等待该数据的重新传输

把在TCP传输过程中，由于单个数据包的丢失而造成的阻塞称为TCP上的队头阻塞


我们知道在 HTTP/2 中，多个请求是跑在一个 TCP 管道中的，如果其中任意一路数据流中出现了丢包的情况，那么就会阻塞该 TCP 连接中的所有请求。这不同于 HTTP/1.1，使用 HTTP/1.1 时，浏览器为每个域名开启了 6 个 TCP 连接，如果其中的 1 个 TCP 连接发生了队头阻塞，那么其他的 5 个连接依然可以继续传输数据

HTTP 是什么？

HTTP: 超文本传输协议 –> 拆成三个部分： 超文本、传输、协议
协议： 一种约束、规范，遵守协议则就要遵守这些约束和规范
传输： A 传给 B，B 传给 A。HTTP 是双向协议。 这里恰好可以用客户端、服务端的数据传送来说明
超文本：所谓超文本，字面意思就是超越文本的文本。它是文字、图片、视频、音乐等多媒体的混合体，并且它拥有超链接。能够从一个超文本跳跃到另一个超文本

为什么HTTP要设置成无状态的

无状态是为了降低服务端的压力，避免http连接长期占用服务器资源。对于服务器端的缓存、负载均衡都有一定的好处

URI 和 URL的区别？

URI 和 URL 都能够指定唯一的一个资源。但是URI的宗旨是 无论你用的是什么方式，只要你能唯一的确定一个资源，就都算URI的一种实现，而URL就是用定位的方式(…/…/…/)来定位具体的资源
URI = URL + URN

爬虫

HTTP协议没有规定用户代理后面必须是「真正的人类」，它也完全可以是「机器人」，这些「机器人」的正式名称就叫做爬虫，实际上是一种可以自动访问Web资源的应用程序。互联网上至少有 50% 的流量都是由爬虫产生的，如果你的网站今天的访问量是十万，那么里面至少有五六万是爬虫机器人，而非真实的用户。
绝大多数的爬虫都是由各大搜索引擎「放」出来的，抓取网页存入庞大的数据库，再建立关键字索引，这样我们才能够在搜索引擎中快速地搜索到互联网角落里的页面。

TCP/IP是有多种协议共同构成的一个协议栈
这个协议栈有4层：
应用层

链接层：负责在以太网、WiFi这样的底层网络上发生原始数据包，使用MAC地址来标记网络上的设备(也叫MAC层)

DNS的作用就是通过域名找到对应的IP地址
查找顺序：浏览器 操作系统 host文件 | DNS根服务器 一级域名 二级域名 三级域名 返回客户端 —> 这是递归查询
（客户端电脑使用的DNS就是本地DNS服务器）

  1. 本地客户端向服务器发起请求查询IP地址 (这里是向本地DNS服务器发起请求过程中进行下面的2 3 4 步)
  2. 查看浏览器有无该域名的IP缓存
  3. 查看操作系统有无该域名的IP缓存
  4. 查看Host文件有无该域名的解析配置
  5. 前面都没有找到IP地址的情况下，会去DNS根服务器查询，会找出对应的一级域名 比如.com的服务器
  6. 在一级域名服务器找 比如google.com这个二级域名服务器
  7. 接下来查找www.google.com这个三级域名的地址
  8. 找到后返回给客户端

注意一个问题： host 到 本地DNS服务器 的查询方式是递归查询 本地DNS服务器 到 根域名服务器等后面的域名服务器的查询方式是迭代


递归查询时，返回的结果只有两种：查询成功或查询失败
迭代查询，返回的是最佳的查询点或者主机地址
这里的本地域名服务器是咋用的？ 应该就是本地客户端向本地服务器发起请求查询IP地址

片段标识符 这个是牛爷爷

它是 URI 所定位的资源内部的一个 锚点 或者说是 标签，浏览器可以在获取资源后直接 跳转到它指示的位置。但片段标识符仅能由浏览器这样的客户端使用，服务器是看不到的。也就是说，浏览器永远不会把带 #fragment 的 URI 发送给服务器，服务器也永远不会用这种方式去处理资源的片段

三次握手

最开始两端都为CLOSED状态。在通信前双方都会建立TCB，服务器创建完TCB后便会进入LISTEN状态，就可以等待客户端发送数据了

  • 第一次握手
    • 客户端向服务器发送连接请求报文段(SYN)。这个报文段中是包含自身的数据通讯初始序号。 发送请求后，客户端变进入了SYN-SENT状态
  • 第二次握手
    • 服务端收到连接请求报文段后，如果是同意连接，则会发送一个应答(ACK + SYN)报文段。这个应答中也会包含自身的数据通讯初始序号，发送完数据后，服务端就进入SYN-RECEIVED状态
  • 第三次握手
    • 当客户端收到服务端发送的同意应答后，向服务端发送一个确认报文(ACK)。客户端发完这个报文段后就进入ESTABLISHED状态，此时建立成功了

SYN报文：连接请求报文 SYN = 1 / ACK = 0; 表示这是一个连接请求报文
ACK报文：同意连接报文

序号：seq (浏览器初始化x 服务器初始化y)
确认号：ack (ack = x + 1, ack = y + 1) 即在对方发送的序号基础上+1

为什么要三次握手呢？
这三次握手恰好就能让双方都能确认当前对方的网络状态如何，防止失效的连接请求报文段被服务端接受，造成错误
关键字：TCP的超时重传机制 服务端一直等待，造成资源浪费

四次挥手

FIN报文， FIN = 1表示此报文段的发生发请求释放运输连接
ACK报文， ACK = 1表示确认要求

  • 第一次握手
    • 客户端如果认为数据发送完成，则它就需要向服务端发送连接释放请求(发送FIN报文段)
  • 第二次握手
    • 服务器 收到连接释放请求后，会告诉应用层要释放 TCP 链接。然后会发送 ACK 包，并进入 CLOSE_WAIT 状态，表示 客户端 到 服务端 的连接已经释放，不接收 客户端 发的数据了。但是因为 TCP 连接时双向的，所以 服务端 仍旧可以发送数据给 客户端
  • 第三次握手
    • 服务端把还没有发完的数据发送给客户端，完成后会向客户端发送请求连接释放请求，然后服务端便会进入LAST-ACK状态
  • 第四次握手
    • 客户端收到释放请求后，向服务器发送应答确认，此时A进入TIME-WAIT状态，这状态会持续2MSL(最大段生存期，指报文段在网络中生存的时间，超时则会被抛弃)，若该时间段内没有服务端的重发请求就进入CLOSED状态，B收到应答确认后就会进入CLOSED 状态
    • (因为发送报文和半关闭状态是不冲突的)

为什么需要经过时间等待计时器设置的时间 2MSL（一个报文的来回时间） 后才会进入 CLOSED 状态？
（这样做的目的是确保服务端收到自己的 ACK 报文。如果服务端在规定时间内没有收到客户端发来的 ACK 报文的话，服务端会重新发送 FIN 报文给客户端，客户端再次收到 FIN 报文之后，就知道之前的 ACK 报文丢失了，然后再次发送 ACK 报文给服务端）

为什么要四次握手呢？
两次握手就可以释放一端到另一端的TCP连接，但是完全释放连接需要四次握手

可靠传输的基本原理

  • 停止等待协议

    发送方发送消息后，需要等待接收方有反馈之后，才会继续发送新的消息：即停止-等待-停止-等待
    但上述是理想情况

下面如果有差错的情况：
超时重传：发送方发送消息后，迟迟没有等待到接收方的确认消息。发送方等待一段时间后会重新发送消息。

  • 发送的消息在路上丢失了

  • 确认的消息在路上丢失了

  • 确认的消息很久才到
    (停止等待协议通过超时重传保证可靠传输)
    超时定时器：每发送一个消息，都需要设置一个定时器
    注意：停止等待协议对信道的利用效率不高

  • 连续ARQ协议(在停止等待协议的基础上进行改造的)

  • ARQ(自动重传请求) —> 单个发送和确认效率低，则进行批量发送和确认

累计确认：只要收到某个消息的确认消息，则表示这一个消息之前的消息我都已经收到了(减少收到确认消息报文的数量，以此提升网络效率)

TCP的流量控制(TCP特有的功能，UDP这些协议是没有这个的)

所谓的流量控制就是发送方传数据传得太快而接收方还来不及接受这么多的数据。利用滑动窗口机制可以很方便的在TCP连接上实现对发送方的流量控制。

UDP 和 TCP 的区别(优劣点)

UDP:

  • 连接不可靠，不能传送大文件
  • 速度比TCP更快
  • 头部开销小:UDP头部8个字节，TCP头部20个字节
  • UDP不止支持一对一传输，还可以支持一对多、多对多、多对一的方式

TCP:

  • 面向连接的(三次握手、四次挥手)、可靠的、基于字节流(UDP是基于数据报，而TCP是基于字节流)的传输层协议

HTTPS

比较好的一篇文章
HTTP是明文传输。如果在浏览器和服务器之间传输数据的时候，有中间人来 窃取/伪造/篡改 数据，就会出现问题。比如我们常见的WiFi钓鱼路由器。这是缺乏安全性的。

HTTPS 本身并没有什么惊世骇俗的本事，全是靠着后面的 SSL/TLS 撑腰(TLS/SSL 是在TCP和HTTP之间插入的安全层，即先要通过SSL才能到达TCP协议)


HTTPS协议提供了三个关键的指标

  • 加密：数据不会被窃取 –> 混合加密
  • 数据一致性：数据不会被篡改 –> 摘要算法
  • 身份认证：知道发送方的真实身份 –> 签名算法(认证)

HTTPS的握手过程

TLS

HTTPS最重要的组成部分就是TLS协议了，因为是这个协议保证了安全性
TLS是SSL的后续版本，是一种在两台设备之间用户身份验证和加密的一种协议
TLS(SSL)是一个独立的协议，不只有HTTP可以使用，其他应用层协议也可以使用，如SMTP(电子邮件协议)、Telnet(远程登录协议)等都可以使用

结合了密钥交换算法、签名算法、对称加密算法、摘要算法等组成的一个密码串

对称加密

加密和解密都使用的是相同的钥匙。
如果单单使用对称加密，这个密码在传输过程中是明文的中间人仍然可以获取到密码

非对称加密

这个涉及公钥 私钥
非对称加密算法的设计要比对称算法难得多

公钥和私钥

公钥和私钥是多对一的关系
注意：公钥加密过的信息，只有私钥能解开；私钥加密过的信息，所有和它有关系的公钥都能解开
证书能够确认 这个公钥是谁发的(避免中间人乱给公钥)

你还是以明文的方式传递公钥，就算被中间人得到也没关系，因为解密公钥只能用私钥。而服务器里藏有私钥，其他人都无法知道

存在的问题：

  • 非对称加密的效率比较低：解密的速度慢，影响用户打开页面的速度
  • 仍然不够安全：中间人可以用公钥加密数据发送给服务器，服务器能够用私钥解开

混合加密：非对称加密 结合 对称加密

TSL使用的是混合加密(因为非对称加密的运算速度很慢，对称加密速度快)
具体流程如下：
发送方使用公钥加密 对称的密钥，接收方收到后，利用私钥解密从而获取 对称的密钥

加密过程
首先浏览器会给服务器发送一个client_random和一个加密的方法列表

服务器接受后给浏览器返回另一个随机数server_random和加密方法

现在两者拥有三种相同的凭证: client_random, server_random和加密方法

接着用这个加密方法将两个随机数混合起来生成密钥，这个密钥就是浏览器和服务器通信的暗号了

摘要算法

MD5 最常用于验证文件的完整性
你可以把摘要算法理解成一种特殊的压缩算法，它能够把任意长度的数据压缩成一种固定长度的字符串，这就好像是给数据加了一把锁
就能够实现数据的完整性，哪怕你在文件中改变一个标点符号，增加一个空格，生成的文件摘要也会完全不同

添加数字证书

为了解决非对称加密中公匙来源的不安全性。我们可以使用数字证书和数字签名来解决。
需要服务器向浏览器证明“我就是我”
让权威机构颁发证书
服务器不是将公钥给浏览器，而是给的数字证书，而公钥就放在数字证书里面
浏览器会对证书进行验证

“数字签名”用于防止数字证书被篡改

CA
CA的全称(Certificate Authority)，证书认证机构，你必须让CA颁布具有认证过的公钥，才能解决公钥的信任问题
全世界具有认证的 CA 就几家

数字证书的申请和验证

服务器 向CA申请证书，在其过程需要提供公钥(私钥自己留着)，所在公司，站点等信息，CA会去审核你的信息是否合法等操作，如果审核通过，就会向极客时间签发认证的数字证书，包含了原本的公钥、序号以及CA生成的签名等内容

浏览器会读取证书内容验证证书的合法性。

WebSocket

websocket 是基于TCP上全双工通信的协议 也就是一个持久化的网络通信协议(服务器也可以主动推送信息给客户端，进行双向的数据传输，所以只需一次握手就可以进行通信了，http是要三次握手才行)
http 是一个半双工的协议(同一时刻，只能单向传输数据)

HTTP/1.1是当前网站主流的协议。HTTP协议是无状态的，所以每一次的请求都是被当成独立的，唯一的。有时候，这样重复的发送请求就会造成冗余的数据传输(例如重复的cookie、session id)。
对于频繁的数据交互来说，你使用HTTP协议效率低下且会有很多冗余的数据进行发送传输，并且你还只能从客户端向服务器主动发送数据，而服务器不能(没有权限)主动向客户端发送数据，只有客户端主动去找服务器，这样即使服务器的数据发生了更改，也无法及时的告诉客户端发生了什么变化

当然有http轮询请求(隔一段时间发送一次请求)来克服上述问题，但始终效果不太好，非常占用服务器资源

WebSocket 解决了 HTTP 不适用于实时通信的这个问题
只要客户端和服务器建立起了连接，就能够实时的进行双向数据通信

WebSocket特点

  • 建立在TCP协议之上，服务端要想实现这个websocket也非常容易
  • 与HTTP协议有良好的兼容性，默认端口也是80和443，并且握手阶段采用HTTP协议
  • 数据格式(二进制、文本)比较轻量，性能开销小，通讯高效
  • 可以发送文本，也可以发送二进制
  • 没有同源限制，客户端可以与任意服务器进行通信

类似的技术：WebRTC(直播就是用的这个技术)

WebSocket通信原理

握手阶段采用的是HTTP协议
发送的请求依然是HTTP请求

Connection: keep-alive, Upgrade –> 告诉服务器要建立升级的协议
Upgrade: websocket –> 升级为websocket

一次HTTP请求后，只要响应头回到浏览器就已经建立好连接了(已经升级为WebSocket协议了)
注意Sec-WebSocket-Key 和 Sec-WebSocket-Accept
Sec-WebSocket-Key的哈希值用来表示这是哪一个客户端(未来的客户端可能会很多)
Sec-WebSocket-Accept的哈希值用来表示这是哪一个服务端

常见状态码

2xx 成功

  • 200 OK， 表示从客户端发来的请求在服务器端被正确处理
  • 204 No Content, 表示后端处理成功，但是并不需要什么实体数据
  • 206 进行范围请求，只会获取一部分的资源

3xx 重定向

  • 301 永久重定向(配合location，浏览器自动处理)
  • 302 临时重定向(配合location，浏览器自动处理)
  • 304 跟协商缓存相关，表示服务端资源未过期

4xx 客户端错误

  • 400 前端发送了一个不合法的json文件，后端无法解析
  • 401 表示未授权，需要发送HTTP请求并带有认证信息 (token失效的时候就会显示)
  • 403 服务器拒绝该请求的访问
  • 404 在服务器中没有找到该请求的资源
  • 405 表示请求的方式不对

5xx 服务端错误

  • 500 表示服务器端在执行请求时发生了错误
  • 501 表示服务器不支持当前请求所需要的某个功能
  • 503 表明服务器暂时处于繁忙/维护状态
